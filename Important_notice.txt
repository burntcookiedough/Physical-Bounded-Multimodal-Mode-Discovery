## Why K-Means Fails for Your Use Case

K-means has four hard assumptions that all break in sensor data from physical systems: [iasj.rdd.edu](https://iasj.rdd.edu.iq/journals/uploads/2025/01/31/c9e44b5695a1c90aad322e48a6568d4f.pdf)

- **Assumes spherical clusters** — operational modes in feature space are almost never round. A motor at high load with high vibration has an ellipsoidal, correlated cluster shape (when current increases, temperature increases too — a non-spherical relationship)
- **Requires you to specify k in advance** — you do not know how many operational modes exist. That is literally the purpose of your project
- **Treats all data points as belonging to a cluster** — real sensor data has noise, transients, and genuine outlier windows. K-means forces every point into a cluster, polluting your mode centroids with garbage
- **Cannot handle clusters of unequal density** — high-load mode may have sparse data (rarely encountered) while idle mode has dense data (most of the time). K-means distorts both

For IoT and industrial sensor datasets specifically, DBSCAN outperforms k-means on silhouette score (0.76 vs lower), handles noise natively, and discovers non-convex cluster shapes. [rmets.onlinelibrary.wiley](https://rmets.onlinelibrary.wiley.com/doi/10.1002/joc.70239)

## Algorithm Comparison for Your Project

| Algorithm | How It Works | Best For | Problem With Your Data |
|---|---|---|---|
| **K-Means** | Assigns points to nearest centroid, re-computes centroids, repeats | Small, spherical, equal-density, known-k datasets | Fails all 4: non-spherical modes, unknown k, noisy sensor data, unequal densities  [iasj.rdd.edu](https://iasj.rdd.edu.iq/journals/uploads/2025/01/31/c9e44b5695a1c90aad322e48a6568d4f.pdf) |
| **DBSCAN** | Groups densely connected points; labels sparse points as noise | Noise-heavy data, unknown k, non-convex shapes | Struggles with clusters of varying density (your modes differ greatly in how often they occur)  [sciencedirect](https://www.sciencedirect.com/science/article/abs/pii/S0009281924000187) |
| **HDBSCAN** | Hierarchical DBSCAN; handles varying densities by building a cluster tree and finding stable regions | **Best all-around for sensor data** — noise-robust, unknown k, varying density, non-convex | Slightly harder to interpret than GMM for physical meaning  [jatit](http://www.jatit.org/volumes/Vol103No10/16Vol103No10.pdf) |
| **GMM (Gaussian Mixture Model)** | Models each cluster as a Gaussian distribution; gives probabilistic soft assignments | Overlapping modes, when you want a confidence probability per assignment | Assumes Gaussian shape (still too rigid for vibration data); requires knowing k; expensive EM training  [datajourney24.substack](https://datajourney24.substack.com/p/advanced-clustering-algorithms-dbscan) |
| **OPTICS** | Ordering-based density clustering; produces a reachability plot revealing cluster hierarchy | Complex, multi-scale density data | Very slow on large datasets; reachability plot hard to automate  [sciencedirect](https://www.sciencedirect.com/science/article/abs/pii/S0009281924000187) |
| **Spectral Clustering** | Converts to graph, finds densely connected subgraphs | Non-convex, manifold-structured data | Scales very poorly — O(n³) — unusable on sliding-window sensor streams of thousands of windows  [jatit](http://www.jatit.org/volumes/Vol103No10/16Vol103No10.pdf) |
| **Agglomerative / Ward** | Bottom-up hierarchical merging | When you want a dendrogram showing mode relationships | Produces full hierarchy but doesn't scale past ~10k points  [jatit](http://www.jatit.org/volumes/Vol103No10/16Vol103No10.pdf) |

## What You Should Actually Use

The answer is **different algorithms at different tiers of your architecture**, chosen strategically. This also **strengthens your patent** because algorithm heterogeneity in your pipeline is itself a documented design choice.

### Tier 1 — Per-Modality Micro-Clusterers (Direction 1)

Use **HDBSCAN** for each modality independently. [jatit](http://www.jatit.org/volumes/Vol103No10/16Vol103No10.pdf)

**Why:**
- Discovers k automatically — you do not need to guess how many vibration modes or current modes exist
- Explicitly marks noisy/transitional windows as outliers (label = -1) — these become your transition-zone candidates
- Handles varying cluster density — idle mode (dense, frequent) and overload mode (sparse, rare) are both discovered correctly
- Silhouette-equivalent metric (DBCV — Density-Based Clustering Validation) gives you per-point confidence scores for your consensus arbiter

```python
import hdbscan

# Per-modality micro-clustering with HDBSCAN
for modality, features in [('vibration', feats_V), ('current', feats_I), ('temperature', feats_T)]:
    clusterer = hdbscan.HDBSCAN(
        min_cluster_size=15,        # minimum windows to form a mode
        min_samples=5,              # core point threshold (noise resistance)
        metric='euclidean',
        cluster_selection_method='eom'  # excess of mass — stable modes
    )
    labels = clusterer.fit_predict(features)
    probabilities = clusterer.probabilities_  # per-point confidence (0 to 1)
    # probabilities replace silhouette scores in your consensus arbiter
    # outlier_scores = clusterer.outlier_scores_  (higher = more anomalous)
```

The `probabilities_` output from HDBSCAN directly replaces silhouette scores in your consensus arbiter — they already represent "how strongly this point belongs to its cluster" on a 0–1 scale, which is even cleaner than silhouette.

### Tier 2 — Joint Mode Discovery (after Consensus)

Use **GMM (Gaussian Mixture Model)** on the consensus joint feature space — but only after HDBSCAN has already defined the structure. [datajourney24.substack](https://datajourney24.substack.com/p/advanced-clustering-algorithms-dbscan)

**Why:** Once HDBSCAN has removed noise and given you roughly k modes, GMM fits smooth Gaussian distributions to each mode. This gives you:
- **Soft probabilistic assignments** — a window can be 70% Mode A and 30% Mode B, which encodes transitional behavior
- **Ellipsoidal covariance** — captures the correlated nature of current + temperature relationship within a mode
- **BIC/AIC score** to validate number of components (confirmatory use after HDBSCAN discovery)

```python
from sklearn.mixture import GaussianMixture

# Use HDBSCAN-discovered k as initialization for GMM
k_discovered = len(set(hdbscan_labels)) - (1 if -1 in hdbscan_labels else 0)

gmm = GaussianMixture(
    n_components=k_discovered,
    covariance_type='full',       # full covariance = ellipsoidal clusters
    n_init=5,                     # multiple restarts for stability
    random_state=42
)
joint_proba = gmm.fit_predict(consensus_features)
soft_assignments = gmm.predict_proba(consensus_features)  # shape: (n_windows, k)
```

### Tier 3 — Physics Feasibility Filter (Direction 2)

The physics filter operates on GMM centroids (means) and covariance matrices — this is actually **better than filtering k-means centroids** because:
- GMM gives you a full covariance matrix per mode, so you can test whether the mode's spread in physical space violates constraints (not just the centroid)
- You can compute the probability mass outside the feasible physical region for each mode — a more precise incoherence measure than binary pass/fail

## The Upgraded Algorithm Pipeline

```
Raw Sensor Windows
       │
       ▼
Per-Modality Feature Extraction (Time + Frequency domain)
       │
       ▼
PER-MODALITY HDBSCAN                    ← replaces k-means
  • Outputs: (label, probability) per point
  • label = -1 → noise/transition zone
  • probability = confidence weight for consensus
       │
       ▼
WEIGHTED CONSENSUS ARBITER              ← Direction 1 (unchanged)
  • Uses HDBSCAN probabilities as weights (not silhouette)
  • Conflict flag fires when labels disagree
  • Outlier windows (label = -1 in majority) → flagged as TRANSITION
       │
       ▼
JOINT GMM on non-noise consensus points ← replaces joint k-means
  • k_init = HDBSCAN-discovered mode count
  • Full covariance captures cross-modal correlations
  • Soft assignments encode transitional windows
       │
       ▼
PHYSICS FEASIBILITY FILTER              ← Direction 2 (now tests GMM means + covariance)
  • Test mode centroid against Joule heating, power balance, etc.
  • Test covariance bounds: does mode spread into physically impossible region?
       │
       ▼
VALIDATED OPERATIONAL MODES + DIAGNOSTICS
```
