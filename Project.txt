## PART 1: THE BASELINE PROJECT

### 1.1 Project Objective (What Are We Building?)

We are building a **zero-label operational mode discovery system** for physical equipment.

**Problem:** Industrial equipment (motors, pumps, compressors, HVAC systems) operates in multiple behavioral regimes. Engineers traditionally had to manually define these regimes ("normal operation," "stressed," "failing"). This is time-consuming, requires expertise, and misses emergent behaviors.

**Solution:** Our system automatically discovers these operational modes by observing multimodal sensor data in an unsupervised fashion. No labels needed.

**Key Characteristic:** The system learns *what operational modes exist* purely from data patterns, without any human annotation. If vibration, electrical current, and temperature naturally cluster into K distinct groups, the system identifies those K modes and characterizes them.

### 1.2 Hardware Testbed

We collect real sensor data from a physical system such as:
- Three-phase induction motor

                       │

┌──────────────────────────────────────────────────────────┐
│ STEP 2: FEATURE EXTRACTION & TRANSFORMATION              │
│                                                            │
│ From raw sensor signals, we compute engineered features:  │
│                                                            │
│ TIME-DOMAIN FEATURES:                                    │
│   • RMS (Root Mean Square): overall energy level          │
│   • Peak: maximum value in window                         │
│   • Crest Factor: peak / RMS (impulse indicator)          │
│   • Skewness: asymmetry of distribution                   │
│   • Kurtosis: severity of impulses                        │
│                                                            │
│ FREQUENCY-DOMAIN FEATURES:                               │
│   • FFT (Fast Fourier Transform): frequency content       │
│   • Power Spectral Density (PSD): energy per frequency    │
│   • Harmonic peaks: dominant frequency components         │
│   • Spectral entropy: randomness of spectrum              │
│   • Spectral centroid: mass center of spectrum            │
│                                                            │
│ STATISTICAL FEATURES:                                    │
│   • Entropy: information content of signal                │
│   • Quantiles: percentile values (25%, 50%, 75%)          │
│   • Variance: dispersion around mean                      │
│                                                            │
│ CROSS-MODAL FEATURES (between sensors):                   │
│   • Time-lag correlation: how vibration leads current     │
│   • Phase alignment: synchronization between modalities    │
│   • Cross-correlation peaks: causality hints              │
│                                                            │
│ Result: Each time window (e.g., 1 second of data)        │
│ is transformed into a feature vector:                     │
│ [rms_vib, peak_vib, entropy_vib, ..., rms_curr, ...,     │
│  rms_temp, ..., corr_vib_curr, lag_vib_curr, ...]        │

└──────────────────────┬───────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────┐
│ STEP 3: NORMALIZATION & PREPROCESSING                    │
│                                                            │
│ Features are standardized to make them comparable:        │
│                                                            │
│ Z-SCORE STANDARDIZATION:                                 │
│   For each feature f, compute:                            │
│   f_normalized = (f - mean(f)) / std(f)                   │
│   Result: mean=0, std=1 for each feature dimension        │
│                                                            │
│ OUTLIER HANDLING:                                        │
│   • IQR (Interquartile Range) clipping: remove extreme    │
│     values (< Q1-1.5×IQR or > Q3+1.5×IQR)                │
│   • Percentile bounds: cap at 1st and 99th percentile     │
│                                                            │
│ MISSING VALUE IMPUTATION:                                │
│   • Forward-fill: use last known value                    │
│   • Interpolation: linear fit between neighboring values   │
│   • Mean imputation: use feature mean (last resort)       │
│                                                            │
│ Result: Clean, standardized feature matrix                │
│ Dimensions: N samples × M features (M = 40-80)            │
└──────────────────────┬───────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────┐
│ STEP 4: UNSUPERVISED CLUSTERING                          │
│                                                            │
│ Apply clustering algorithm to discover natural groupings: │
│                                                            │
│ K-MEANS CLUSTERING:                                      │
│   1. Choose k (number of clusters)                        │
│   2. Randomly initialize k centroids                      │
│   3. Iterate:                                             │
│      • Assign each point to nearest centroid              │
│      • Update centroids as mean of assigned points        │
│   4. Stop when centroids converge                         │
│   Output: cluster labels, centroid locations              │
│                                                            │
│ DBSCAN (Density-Based Clustering):                        │
│   1. Define density neighborhood (eps, min_samples)       │
│   2. Points with neighbors → form clusters                │
│   3. Isolated points → outliers/noise                     │
│   Output: cluster labels, noise flags                     │
│                                                            │
│ GAUSSIAN MIXTURE MODEL (GMM):                             │
│   1. Model each cluster as Gaussian distribution          │
│   2. EM algorithm finds optimal mixing proportions        │
│   3. Output: cluster probabilities (soft assignment)      │
│                                                            │
│ Result: Each data point is assigned to an operational     │
│ mode (cluster label): 0, 1, 2, ..., K-1                  │

└──────────────────────┬───────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────────┐
│ STEP 5: CLUSTER VALIDATION & INTERPRETATION              │
│                                                            │
│ Assess quality of discovered modes:                       │
│                                                            │
│ SILHOUETTE COEFFICIENT:                                  │
│   For each point, measure:                                │
│   • a = avg distance to points in same cluster            │
│   • b = avg distance to points in nearest cluster         │
│   • silhouette = (b - a) / max(a, b)                      │
│   Range: [-1, +1]                                         │
│   Interpretation:                                         │
│     +1 → point is very well clustered                     │
│      0 → point is ambiguous                               │
│     -1 → point may be misassigned                         │
│                                                            │
│ DAVIES-BOULDIN INDEX:                                    │
│   Compares intra-cluster vs inter-cluster distances       │
│   Lower value = better separation                         │
│   Formula involves avg intra-cluster variance             │
│   and inter-cluster distances                             │
│                                                            │
│ CALINSKI-HARABASZ SCORE:                                 │
│   Ratio of between-cluster variance to within-cluster     │
│   Higher = better defined clusters                        │
│                                                            │
│ HUMAN ANNOTATION (Optional):                              │
│   Domain expert manually labels discovered modes:         │
│   "Cluster 0 = Normal operation"                          │
│   "Cluster 1 = High mechanical load"                      │
│   "Cluster 2 = Thermal stress"                            │

## PART 2: WHY BASELINE IS NOT PATENTABLE

### 2.1 The Patent Novelty Problem

**Question:** Can we file the baseline project as-is for a patent?

**Answer:** No. It will be **rejected for lack of novelty and obviousness**.

### 2.3 Patent Law: Why Baseline Fails

Two legal tests determine patentability:

**Test 1: Novelty (35 U.S.C. §102)**

*Standard:* An invention is NOT novel if a single prior-art patent or publication teaches every element of your claim.

*Baseline Status:* **FAILS**

IBM's patent (US11277425B2) teaches:
- Multimodal sensor collection ✓
- Unsupervised clustering ✓
- Feature extraction & normalization ✓
- Cluster validation metrics ✓
- State-transition graph generation ✓

A patent examiner will cite this single patent and state: *"Your baseline project is anticipated by US11277425B2. Every element is disclosed."*

**Test 2: Non-Obviousness (35 U.S.C. §103)**

*Standard:* Even if no single patent teaches everything, your combination must not be obvious to a person skilled in the art.

*Baseline Status:* **FAILS**

A skilled practitioner in machine learning would find this combination obvious:
- It's routine practice to extract multiple types of features from sensor data
- Applying k-means/DBSCAN to extracted features is standard
- Computing Silhouette scores for validation is textbook practice
- Building a state-transition graph from cluster labels is straightforward

A patent examiner will state: *"Combining existing clustering algorithms with standard validation metrics is obvious. Your application merely applies known techniques in a predictable manner."*

### 2.4 The "Combination Rejection"

Examiners often cite multiple prior-art patents and argue their combination is obvious:

Rejection Logic:
─────────────────────────────────────────────────────

Intel (US20210110197A1) teaches:
  → Independent clustering per modality + fusion

+

IBM (US11277425B2) teaches:
  → Unsupervised mode extraction from sensors

+

WO2017058993A1 teaches:
  → Drift detection in sensor streams

=

OBVIOUS COMBINATION (per 35 USC §103):
"It would be obvious to a person skilled in data mining
and cyber-physical systems to apply independent clustering
per modality, validate with standard metrics, and detect
state transitions. This is merely routine engineering."

## PART 4: DIRECTION 1 — HIERARCHICAL CONSENSUS ARCHITECTURE

### 4.1 Core Concept

**Current Baseline Approach:**
All sensors → Concatenate features → Single clustering algorithm → Mode labels

**Direction 1 Approach:**
Vibration → K-means (Cluster 0, 1, 2) + Silhouette scores
Current   → K-means (Cluster 0, 1, 2) + Silhouette scores
Temp      → K-means (Cluster 0, 1, 2) + Silhouette scores
             ↓
          Consensus Arbiter (Weighted Voting)
             ↓
          Joint Mode Label + Modality Conflict Flag

**Key Novelty:** We maintain per-modality cluster assignments as *independent, first-class artifacts*, then use modality-specific confidence (silhouette scores) as weights in a consensus mechanism. When modalities disagree, we **surface that disagreement as an explicit system output** — not hidden inside a blended centroid.

### 4.2 Why This Is Novel

**What Prior Art Does:**
- Intel US20210110197A1: Independent clustering on each modality, then *fuses* outputs for joint decision. Doesn't maintain per-modality identity or surface disagreement.
- IBM US11277425B2: Concatenates all modality features into one vector. No per-modality tracking.

**What Direction 1 Does (New):**
- Maintains per-modality cluster assignments throughout the pipeline
- Computes per-modality **silhouette scores as dynamic confidence weights**
- Implements **weighted consensus voting** where low-confidence modalities contribute less
- Explicitly flags **modality disagreement** as an operational signal
- Derives **modality dominance signature** for each mode (which sensors characterize it best)

**Why It's Non-Obvious:**
1. Different physical sensors respond to the same failure at different speeds and with different noise. Treating them symmetrically (Intel/IBM approach) loses this information.
2. No prior patent maintains per-modality cluster identity through consensus. All either pre-fuse (IBM) or post-fuse (Intel), but don't track disagreement.
3. The insight that modality-specific silhouette scores can serve as *dynamic confidence weights* in consensus, and that disagreement itself is operationally meaningful, is not suggested by prior art.

### 4.3 System Architecture (Direction 1)

INPUT: Multimodal sensor streams
       Vibration (100 Hz, 1 second window = 100 samples)
       Current (1 kHz, 1 second window = 1000 samples)
       Temperature (10 Hz, 1 second window = 10 samples)

STEP 1: FEATURE EXTRACTION (per modality, independent)
        ┌──────────────────────────────────────┐
        │ Vibration Features:                  │
        │ [RMS, Peak, Entropy, FFT_peaks, ...] │ (20 features)
        └──────────────────────────────────────┘
        
        ┌──────────────────────────────────────┐
        │ Current Features:                    │
        │ [RMS, Harmonics, Phase, Crest, ...]  │ (18 features)
        └──────────────────────────────────────┘
        
        ┌──────────────────────────────────────┐
        │ Temperature Features:                │
        │ [Value, Rate_of_change, Trend, ...]  │ (8 features)
        └──────────────────────────────────────┘

STEP 2: NORMALIZATION (per modality, independent)
        Each modality's features standardized to mean=0, std=1
        → vibration_normalized, current_normalized, temp_normalized

STEP 3: PER-MODALITY MICRO-CLUSTERING (PARALLEL, INDEPENDENT)
        
        Vibration Micro-Clusterer:
        ┌─────────────────────────────────────────────────────┐
        │ Input: vibration_normalized (n_samples × 20 features)│
        │ Algorithm: K-Means (k determined via silhouette)     │
        │ Output:                                              │
        │   - label_V: cluster assignment (0, 1, 2, ...)      │
        │   - confidence_V: silhouette_score per point        │
        │   - centroid_V: center of each cluster              │
        │ Example output:                                      │
        │   Sample 1: label=0, confidence=0.78                │
        │   Sample 2: label=1, confidence=0.65                │
        │   Sample 3: label=0, confidence=0.92                │
        └─────────────────────────────────────────────────────┘
        
        Current Micro-Clusterer:
        ┌─────────────────────────────────────────────────────┐
        │ Input: current_normalized (n_samples × 18 features)  │
        │ Algorithm: K-Means (k determined via silhouette)     │
        │ Output:                                              │
        │   - label_I: cluster assignment                      │
        │   - confidence_I: silhouette_score per point         │
        │   - centroid_I: center of each cluster               │
        │ Example output:                                      │
        │   Sample 1: label=1, confidence=0.51                │
        │   Sample 2: label=1, confidence=0.48                │
        │   Sample 3: label=0, confidence=0.89                │
        └─────────────────────────────────────────────────────┘
        
        Temperature Micro-Clusterer:
        ┌─────────────────────────────────────────────────────┐
        │ Input: temp_normalized (n_samples × 8 features)      │
        │ Algorithm: K-Means (k determined via silhouette)     │
        │ Output:                                              │
        │   - label_T: cluster assignment                      │
        │   - confidence_T: silhouette_score per point         │
        │   - centroid_T: center of each cluster               │
        │ Example output:                                      │
        │   Sample 1: label=0, confidence=0.81                │
        │   Sample 2: label=0, confidence=0.72                │
        │   Sample 3: label=1, confidence=0.91                │
        └─────────────────────────────────────────────────────┘

STEP 4: CONSENSUS ARBITRATION (WEIGHTED VOTING)
        
        For each time point (sample), perform consensus:
        
        Inputs:
          label_V, confidence_V (vibration cluster + confidence)
          label_I, confidence_I (current cluster + confidence)
          label_T, confidence_T (temperature cluster + confidence)
          
          weight_V = 0.40  (vibration is most diagnostic for mechanical)
          weight_I = 0.35  (current is diagnostic for electrical)
          weight_T = 0.25  (temperature is slower, less weight)
        
        Consensus Formula (weighted voting):
          votes = {}
          votes[label_V] += confidence_V × weight_V
          votes[label_I] += confidence_I × weight_I
          votes[label_T] += confidence_T × weight_T
          
          joint_mode = argmax(votes)
          consensus_confidence = max(votes) / sum(votes)
        
        Example Calculation:
          Sample 1:
            label_V=0, conf_V=0.78 → votes[0] += 0.78 × 0.40 = 0.312
            label_I=1, conf_I=0.51 → votes[1] += 0.51 × 0.35 = 0.179
            label_T=0, conf_T=0.81 → votes[0] += 0.81 × 0.25 = 0.203
            
            Total: votes[0] = 0.515, votes[1] = 0.179
            joint_mode = 0 (mode with highest votes)
            consensus_confidence = 0.515 / (0.515 + 0.179) = 0.742
        
        Example with CONFLICT:
          Sample 2:
            label_V=0, conf_V=0.78 → votes[0] += 0.312
            label_I=2, conf_I=0.61 → votes[2] += 0.61 × 0.35 = 0.214
            label_T=1, conf_T=0.65 → votes[1] += 0.65 × 0.25 = 0.163
            
            Total: votes[0] = 0.312, votes[1] = 0.163, votes[2] = 0.214
            joint_mode = 0 (plurality)
            consensus_confidence = 0.312 / 0.689 = 0.453 (LOW!)
            
            DETECTION: Modality CONFLICT!
            Vibration says "Mode 0" but Current and Temp disagree.
            → Flag this sample as CONFLICTED
            → Output: MODALITY_CONFLICT_FLAG = TRUE
            → Interpretation: System is transitioning, or something abnormal

STEP 5: MODALITY DOMINANCE SIGNATURE
        
        For each discovered joint mode M:
        
        1. Collect all samples assigned to joint_mode = M
        2. Extract their per-modality confidence scores
        3. Compute average confidence per modality for Mode M:
        
           avg_conf_V[M] = mean(confidence_V for samples in Mode M)
           avg_conf_I[M] = mean(confidence_I for samples in Mode M)
           avg_conf_T[M] = mean(confidence_T for samples in Mode M)
        
        4. Normalize to create dominance signature:
        
           sum_conf = avg_conf_V[M] + avg_conf_I[M] + avg_conf_T[M]
           dominance_V[M] = avg_conf_V[M] / sum_conf
           dominance_I[M] = avg_conf_I[M] / sum_conf
           dominance_T[M] = avg_conf_T[M] / sum_conf
        
        Example:
          Mode 0 (Normal Operation):
            avg_conf_V = 0.82
            avg_conf_I = 0.71
            avg_conf_T = 0.68
            sum = 2.21
            
            dominance_V = 0.82 / 2.21 = 0.371 (37%)
            dominance_I = 0.71 / 2.21 = 0.321 (32%)
            dominance_T = 0.68 / 2.21 = 0.308 (31%)
            
          INTERPRETATION:
            "Normal Operation is best identified by observing VIBRATION
             dynamics (37% dominance). Secondary confirmation from electrical
             load (32%) and temperature (31%)."
          
          Mode 2 (Thermal Stress):
            dominance_V = 0.25 (25%)
            dominance_I = 0.45 (45%)
            dominance_T = 0.30 (30%)
            
          INTERPRETATION:
            "Thermal Stress is best identified by CURRENT dynamics (45%),
             suggesting electrical fault or overload. Temperature is less
             discriminative (30%), vibration least (25%)."

OUTPUT:
  joint_mode_labels: array of final mode assignments
  modality_conflict_flags: boolean array (TRUE if modalities disagreed)
  modality_dominance_signatures: dict mapping mode → (domV, domI, domT)
  consensus_confidences: array of consensus_confidence per sample

### 4.4 Why This Matters Physically

Different sensors reveal different failure modes:

Scenario 1: Motor Bearing Wear
  Vibration sensor: INCREASES (bearing plays generate impulses)
  Current sensor: slight INCREASE (friction losses)
  Temperature: INCREASES slowly (friction generates heat)
  
  Direction 1 detects: Vibration cluster confident, Current/Temp less so
  → Dominance signature shows HIGH vibration dominance
  → ALERT: "This is a mechanical wear failure (vibration-driven)"

Scenario 2: Electrical Overload
  Vibration sensor: stays NORMAL (no mechanical issue)
  Current sensor: SPIKES (high draw)
  Temperature: INCREASES quickly (Joule heating)
  
  Direction 1 detects: Current/Temp clusters confident, Vibration less so
  → Dominance signature shows HIGH current dominance
  → ALERT: "This is an electrical fault (load-driven)"

Scenario 3: Transition / System Changing
  Vibration: moving toward cluster 1 (high-load cluster)
  Current: still in cluster 0 (normal load)
  Temperature: between clusters (ambiguous)
  
  Direction 1 detects: Modality CONFLICT (tri-modal disagreement)
  → CONFLICT_FLAG = TRUE
  → ALERT: "Modalities disagree → system is transitioning or faulty"
  → Early warning before full transition

## PART 5: DIRECTION 2 — PHYSICS-CONSTRAINED CLUSTER VALIDATION

### 5.1 Core Concept

**Current Baseline Approach:**
Clustering → Silhouette validation (statistical) → Accept all clusters as valid modes

**Direction 2 Approach:**
Clustering → Physics Feasibility Filter → Accept ONLY physically coherent clusters
                                       → Reject/refine physically impossible clusters

**Key Novelty:** After clustering discovers modes statistically, a post-processing **physics feasibility filter** evaluates each cluster centroid against parametric physical constraint inequalities. Clusters whose centroids violate physical laws (Joule heating, thermal inertia, power balance, etc.) are flagged as **physically incoherent** and either rejected, merged, or have their boundaries adjusted.

No prior patent merges unsupervised clustering with parametric physics constraints.

### 5.2 Why This Is Novel

**What Prior Art Does:**
- IBM US11277425B2: Validates clusters using statistical metrics (Silhouette, BIC). No physics constraints.
- WO2017058993A1: Detects drift in sensor streams but doesn't filter clusters by physical feasibility.
- All existing patents treat clustering and physics as separate domains.

**What Direction 2 Does (New):**
- Injects parametric physical constraint inequalities (derived from equipment datasheets) as a **post-clustering validation layer**
- Uses constraint violations to identify and reject physically incoherent clusters
- Provides explicit constraint-violation logs that encode root-cause information
- Demonstrates that **physics constraints can substitute for supervision** in unsupervised learning

**Why It's Non-Obvious:**
1. **Cross-Domain Insight:** The gap between data mining (purely statistical) and physics modeling (equation-based) has been treated as unbridgeable in unsupervised learning. Using *parametric physical constraints* as a validation filter is structurally novel.
2. **Practical Insight:** Phantom clusters created by statistical artifacts can be eliminated by checking against physical laws.
3. **Diagnostic Value:** Constraint violations encode operational meaning (Joule heating violation → electrical problem; thermal inertia violation → sensor malfunction).

### 5.3 Physical Constraint Inequalities

EQUIPMENT PARAMETERS (from motor/pump/compressor datasheet):

R_stator = 2.1 Ω              (stator DC resistance)
I_max = 30 A                  (maximum continuous current)
T_ambient = 25°C              (ambient temperature)
T_max_insulation = 155°C      (max operating temperature, Class B motor)
h_conv = 15 W/(m²K)           (convection heat transfer coefficient)
A_surface = 0.45 m²           (equipment surface area)
τ_thermal = 480 s             (thermal time constant)
a_max = 0.72 g = 7.07 m/s²   (max vibration per ISO 10816)
eta_nominal = 0.88            (nominal motor efficiency)

═══════════════════════════════════════════════════════════════

CONSTRAINT 1: JOULE HEATING FEASIBILITY
─────────────────────────────────────────

Physics: Heat generated in stator resistance = I²R

Constraint Inequality:
  T_measured ≤ T_ambient + (I_rms² × R_stator) / (h_conv × A_surface) + ε

Interpretation:
  For given current draw, there's a maximum temperature the motor can reach
  via Joule heating alone. If cluster centroid exceeds this, it's infeasible
  unless additional heat source (external, mechanical friction) exists.

Example:
  Cluster A centroid has:
    I_rms = 15 A
    T = 95°C
  
  Theoretical max T from Joule:
    T_max = 25 + (15² × 2.1) / (15 × 0.45)
          = 25 + (225 × 2.1) / 6.75
          = 25 + 70
          = 95°C
  
  Centroid is RIGHT AT the physical limit → FEASIBLE (with margin)
  
  Cluster B centroid has:
    I_rms = 10 A
    T = 105°C
  
  Theoretical max T from Joule:
    T_max = 25 + (10² × 2.1) / (15 × 0.45)
          = 25 + (100 × 2.1) / 6.75
          = 25 + 31
          = 56°C
  
  Centroid VIOLATES: T=105°C but T_max=56°C
  → This cluster is PHYSICALLY INCOHERENT
  → Suggests: (a) temperature sensor malfunction, or
              (b) cluster is mixing unrelated data, or
              (c) actual equipment thermal model is wrong (unexpected friction)

═══════════════════════════════════════════════════════════════

CONSTRAINT 2: THERMAL INERTIA (Rate of Change)
───────────────────────────────────────────────

Physics: Motor cannot change temperature instantaneously
  dT/dt = (P_in - P_out) / (m × c_p) ≈ ΔT / τ_thermal

Constraint Inequality:
  |ΔT| / Δt ≤ (P_max / (m × c_p)) = system_dependent_rate
  
  Typical for 5.5 kW motor: max rate ≈ 2°C/s

Interpretation:
  If cluster centroid shows temperature jump that's too fast,
  it's physically impossible.

Example:
  Sample transitions from:
    Time t: Mode_A with T=40°C
    Time t+5s: Mode_B with T=150°C
  
  Rate of change: (150-40) / 5 = 22°C/s
  Physical limit: 2°C/s
  
  VIOLATION: Mode_B is PHYSICALLY INCOHERENT (rate is impossible)
  → Suggests: (a) sensor malfunction (sudden spike), or
              (b) Mode_B doesn't actually exist (statistical artifact)

═══════════════════════════════════════════════════════════════

CONSTRAINT 3: VIBRATION BOUNDS
───────────────────────────────

Physics: Equipment manufacturer specifies maximum acceleration
  ISO 10816 (motor vibration standard):
    Class II equipment: a_max = 0.72 g = 7.07 m/s²

Constraint Inequality:
  a_measured ≤ a_max

Example:
  Cluster C centroid has:
    Vibration RMS: 0.5 g (FEASIBLE, within limit)
  
  Cluster D centroid has:
    Vibration RMS: 1.2 g (INFEASIBLE, exceeds Class II limit)
  
  → Cluster D is either a fault state (acceptable if labeled as such)
    or physically impossible under normal conditions

═══════════════════════════════════════════════════════════════

CONSTRAINT 4: POWER EFFICIENCY
───────────────────────────────

Physics: Energy conservation
  Electrical input power = Mechanical output power + Losses
  η = P_out / P_in

Constraint Inequality:
  η_measured ≤ η_nominal + ε (say, +5% tolerance for measurement error)

Example:
  Cluster E centroid implies:
    η = 0.95 (95% efficiency)
  
  Motor nominal: η = 0.88 (88%)
  
  VIOLATION: η=95% > η_nominal=88% (too good to be true)
  → Suggests: (a) current or power measurement error, or
              (b) cluster mixes data from different operating modes

═══════════════════════════════════════════════════════════════

CONSTRAINT 5: CROSS-MODAL COHERENCE
────────────────────────────────────

Physics: Current load and temperature are causally linked
  dT/dt ∝ P_electrical (load determines heating rate)

Constraint Inequality:
  correlation(dI/dt, dT/dt) ≥ 0.5 (at least moderate positive correlation)

Example:
  Cluster F has data points where:
    Current increases → Temperature decreases (contradicts physics)
    correlation(dI/dt, dT/dt) = -0.3 (negative!)
  
  VIOLATION: Cross-modal incoherence
  → Suggests: (a) one of the sensors is broken, or
              (b) cluster contains unrelated data points

### 5.4 Constraint Validation Algorithm (Direction 2)

ALGORITHM: PHYSICS_FEASIBILITY_FILTER
────────────────────────────────────────

INPUT:
  clusters: dict mapping cluster_id → list of point indices
  centroids: dict mapping cluster_id → feature vector (normalized)
  constraint_params: dict of physical parameters
  features_normalized: full feature matrix

PROCESS:
  for each cluster_id in clusters:
    centroid = centroids[cluster_id]
    
    Step 1: Denormalize centroid back to physical units
      I_rms_physical = centroid[current_feature] × I_sensor_max
      T_physical = centroid[temp_feature] × T_sensor_max
      a_rms_physical = centroid[vibration_feature] × a_sensor_max
    
    Step 2: Test each constraint inequality
      violations = []
      
      // Constraint 1: Joule Heating
      T_max_joule = T_ambient + (I_rms² × R) / (h × A)
      if T_physical > T_max_joule + tolerance:
        violations.append("JOULE_HEATING_VIOLATION")
      
      // Constraint 2: Thermal Inertia
      cluster_data = features[clusters[cluster_id], :]
      rates = compute_temperature_rates(cluster_data)
      if max(rates) > max_physical_rate:
        violations.append("THERMAL_INERTIA_VIOLATION")
      
      // Constraint 3: Vibration Bounds
      if a_rms_physical > a_max:
        violations.append("VIBRATION_BOUND_VIOLATION")
      
      // Constraint 4: Power Efficiency
      eta_measured = compute_efficiency(I_rms_physical, T_physical, ...)
      if eta_measured > eta_nominal + 0.05:
        violations.append("EFFICIENCY_VIOLATION")
      
      // Constraint 5: Cross-Modal Coherence
      corr_IT = compute_correlation(cluster_data, I_feature, T_feature)
      if corr_IT < 0.5:
        violations.append("COHERENCE_VIOLATION")
    
    Step 3: Classify cluster
      if len(violations) == 0:
        status[cluster_id] = "PHYSICALLY_COHERENT"
        valid_clusters.append(cluster_id)
      else:
        status[cluster_id] = "PHYSICALLY_INCOHERENT"
        invalid_clusters.append(cluster_id)
        violation_logs[cluster_id] = violations

OUTPUT:
  valid_clusters: list of clusters that passed all constraints
  invalid_clusters: list of clusters that failed constraints
  violation_logs: dict mapping cluster_id → list of constraint violations
  
  Example:
    Cluster 0: PHYSICALLY_COHERENT (Joule, Thermal, Vibration, Power, Coherence all OK)
    Cluster 1: PHYSICALLY_INCOHERENT (violations: [JOULE, EFFICIENCY])
    Cluster 2: PHYSICALLY_COHERENT
    Cluster 3: PHYSICALLY_INCOHERENT (violations: [THERMAL_INERTIA])

### 5.5 Physical Interpretation of Violations

VIOLATION → ROOT CAUSE MAPPING
═════════════════════════════════════════════════════════════

JOULE_HEATING_VIOLATION
  └─ Temperature exceeds what Joule law permits
     → Could mean:
        1. Current sensor reads too low (actual I is higher)
        2. Temperature sensor reads too high (actual T is lower)
        3. Equipment has additional heat source (mechanical friction)
        4. Stator resistance is higher than expected (aging)

THERMAL_INERTIA_VIOLATION
  └─ Temperature changes too fast (faster than τ allows)
     → Could mean:
        1. Temperature sensor malfunction (noise spike)
        2. Rapid load change (external event, not steady-state)
        3. Thermal model parameters need re-calibration
        4. Different equipment than expected in testbed

VIBRATION_BOUND_VIOLATION
  └─ Vibration exceeds equipment specifications
     → Could mean:
        1. Bearing damage / wear (legitimate fault state)
        2. Vibration sensor malfunction (saturation)
        3. Cluster is mixing normal + abnormal data
        4. Equipment is operating beyond rated conditions

EFFICIENCY_VIOLATION
  └─ Measured efficiency too high (violates conservation of energy)
     → Could mean:
        1. Current sensor malfunction (reading too low)
        2. Power sensor malfunction
        3. Cluster contains data from different load points mixed
        4. Measurement drift over time

COHERENCE_VIOLATION
  └─ Current and temperature don't covary as physics predicts
     → Could mean:
        1. Temperature sensor broken / disconnected
        2. Current sensor broken / disconnected
        3. Thermal model wrong (unexpected cooling source, e.g., fan)
        4. Cluster is poorly separated (contains unrelated regimes)

## PART 6: DIRECTION 3 — ADAPTIVE DATA-RATE GATING

### 6.1 Core Concept

**Current Baseline Approach:**
Continuous sampling at fixed rate → Feature extraction (fixed window) → Clustering

**Direction 3 Approach:**
Online Stability Estimator (Coefficient of Variation)
  ↓
  ├─ CV < threshold → SLOW PATH (1 Hz, long window, FFT-based)
  └─ CV ≥ threshold → FAST PATH (50 Hz, short window, time-domain)
        ↓
    [Reconstruction Error from Clustering as Feedback]
        ↓
    [Transition Manifold Extraction & Pre-Transition Prediction]

**Key Novelty:** The data acquisition and feature extraction pipeline itself becomes **adaptive**, switching between two rates based on system behavior. Reconstruction error from the clustering model feeds back to trigger higher sampling fidelity before the system fully transitions.

### 6.2 Why This Is Novel

**What Prior Art Does:**
- IBM, Intel, all others: Fixed feature extraction pipeline. Sample rate doesn't change.
- WO2017058993A1: Detects drift but doesn't adapt sensing architecture.

**What Direction 3 Does (New):**
- Dynamically modulates **data acquisition rate itself** (not just analysis)
- Uses **reconstruction error as a feedback signal** to trigger fine-grained sampling
- Builds **transition manifold registry** for predictive (not reactive) pre-transition alerts
- Achieves diagnostic capability without constant high-rate sampling (saves memory/power on edge devices)

### 6.3 System Architecture (Direction 3)

STEP 1: ONLINE STABILITY ESTIMATOR (Parallel, Low-Compute)
────────────────────────────────────────────────────────

For each sensor stream, maintain a sliding window (e.g., 5 seconds):
  Coefficient of Variation (CV) = σ(window) / |μ(window)|
  
  if CV < 0.15 (low variability):
    STABILITY_STATE = "STABLE"
    → Activate SLOW PATH
  
  else if CV ≥ 0.15 (high variability):
    STABILITY_STATE = "UNSTABLE"
    → Activate FAST PATH

Physical Interpretation:
  • CV < 0.15: Sensor reading steady (normal operation)
  • CV ≥ 0.15: Sensor reading fluctuating (transition/change likely)

STEP 2: DUAL FEATURE EXTRACTION PATHS
──────────────────────────────────────

SLOW PATH (Stable Periods, Low Sampling Rate):
  ┌─────────────────────────────────────────┐
  │ Sampling: 1 Hz (1 feature per second)   │
  │ Window: 10 seconds (for FFT resolution) │
  │                                         │
  │ Features:                               │
  │ • FFT magnitude at top 5 harmonics      │
  │ • Power spectral density (PSD)          │
  │ • Spectral entropy                      │
  │ • Frequency-weighted centroid           │
  │                                         │
  │ Purpose: Efficient, frequency-domain    │
  │          characterization of steady     │
  │          operational state              │
  │                                         │
  │ Storage: Stable-Mode Feature Store      │
  │          (memory-efficient)             │
  └─────────────────────────────────────────┘

FAST PATH (Unstable Periods, High Sampling Rate):
  ┌─────────────────────────────────────────┐
  │ Sampling: 50 Hz (1 feature per 20 ms)   │
  │ Window: 200 ms (for impulse capture)    │
  │                                         │
  │ Features:                               │
  │ • RMS, Peak, Crest Factor (impacts)     │
  │ • First derivatives (rate of change)    │
  │ • Second derivatives (jerk/acceleration)│
  │ • Cross-channel correlations            │
  │                                         │
  │ Purpose: Detailed time-domain           │
  │          capture of transition          │
  │          signatures and anomalies       │
  │                                         │
  │ Storage: Transition Feature Store       │
  │          (separate, tagged transitional)│
  └─────────────────────────────────────────┘

STEP 3: RECONSTRUCTION ERROR FEEDBACK GATING
─────────────────────────────────────────────

For each incoming feature vector f_t:
  
  1. Find nearest cluster centroid:
      c_i = argmin_i ||f_t - μ_i||
  
  2. Compute reconstruction error:
      Δ_t = ||f_t - μ_i||  (distance to nearest cluster)
  
  3. Threshold check:
      if Δ_t > 2.0 × σ_cluster_i:  (2 standard deviations)
        ERROR_SIGNAL = HIGH
        → Force FAST PATH sampling (if currently in SLOW)
        → System is leaving known clusters → Transition!
  
      else:
        ERROR_SIGNAL = LOW
        → Can continue SLOW PATH sampling

STEP 4: TRANSITION MANIFOLD EXTRACTION
───────────────────────────────────────

During unstable periods (FAST PATH active), store all feature vectors:
  transition_buffer = [f_t0, f_t1, f_t2, ..., f_tn]

After system returns to stability (CV < threshold for >30 seconds):
  
  1. Apply PCA to transition_buffer:
      Extract top 2 principal components
      → Define 2D manifold plane
  
  2. Fit parametric curve (spline) through manifold:
      Encodes: source_cluster → transition_path → target_cluster
  
  3. Store manifold descriptor:
      {
        'source_mode': Cluster_0,
        'target_mode': Cluster_1,
        'path_signature': spline_coefficients,
        'transition_duration': 12.3 seconds,
        'max_reconstruction_error': 2.8,
        'entry_point': feature_vector_at_start,
        'exit_point': feature_vector_at_end
      }

STEP 5: PRE-TRANSITION PREDICTION
──────────────────────────────────

With stored manifold descriptors, when NEW high-error data arrives:
  
  For each known manifold M in registry:
    distance_to_M = min_distance(current_feature_vector, M)
    
    if distance_to_M < ε (near known manifold):
      PREDICTION: "System is following known transition path"
      ESTIMATED_ARRIVAL: target_mode in ~4.2 seconds
      CONFIDENCE: 0.87 (based on manifold similarity)
      
      → GENERATE PRE-TRANSITION ALERT:
        "Detected: transitioning from Normal→HighLoad based on manifold
         similarity (0.87). Predicted arrival: 4.2 seconds. Recommend:
         reduce load or increase cooling."

OUTPUT SIGNALS:
  stability_state: "STABLE" or "UNSTABLE"
  reconstruction_error: distance to nearest cluster
  transition_prediction: (target_mode, arrival_time, confidence)
  feature_extraction_rate: 1 Hz (SLOW) or 50 Hz (FAST)
  diagnostic_alert: pre-transition warning with lead time

### 6.4 What Can Be Patented (Direction 3)

**Method Claim:**
> A method for adaptive-rate multimodal sensor data acquisition in cyber-physical systems comprising: (a) continuously computing online stability metrics (e.g., coefficient of variation) on sensor data; (b) dynamically switching between slow-path and fast-path feature extraction based on stability state; (c) computing reconstruction error from an unsupervised clustering model and using it as a feedback signal to force fast-path sampling when approaching cluster boundaries; (d) extracting transition manifolds from fast-path data collected during mode transitions; and (e) predicting imminent mode changes by comparing incoming feature vectors to stored transition manifolds, generating pre-transition alerts with estimated time to transition.

### 8.2 Code Structure (Pseudo-Python)

# Main pipeline
class OperationalModeDiscovery:
    
    def __init__(self, sensor_data, equipment_params):
        self.sensor_data = sensor_data
        self.eq_params = equipment_params
        self.clusterers = {}  # per-modality
        self.centroids = {}
        self.constraint_violations = {}
    
    # BASELINE
    def baseline_clustering(self):
        # Concatenate features, apply k-means
        all_features = np.hstack([
            self.extract_features('vibration'),
            self.extract_features('current'),
            self.extract_features('temperature')
        ])
        self.kmeans = KMeans(n_clusters=self.optimal_k).fit(all_features)
        return self.kmeans.labels_
    
    # DIRECTION 1
    def direction1_consensus_clustering(self):
        # Independent per-modality clustering
        label_V, conf_V = self._cluster_modality('vibration')
        label_I, conf_I = self._cluster_modality('current')
        label_T, conf_T = self._cluster_modality('temperature')
        
        # Weighted consensus voting
        joint_labels = self._consensus_arbiter(
            [label_V, label_I, label_T],
            [conf_V, conf_I, conf_T],
            weights=[0.4, 0.35, 0.25]
        )
        
        # Modality conflict detection
        conflicts = self._detect_conflicts(label_V, label_I, label_T)
        
        # Dominance signatures
        dominance = self._compute_dominance(joint_labels, conf_V, conf_I, conf_T)
        
        return joint_labels, conflicts, dominance
    
    def _cluster_modality(self, modality_name):
        features = self.extract_features(modality_name)
        features_norm = StandardScaler().fit_transform(features)
        k = self._estimate_optimal_k(features_norm)
        km = KMeans(n_clusters=k).fit(features_norm)
        silhouettes = silhouette_samples(features_norm, km.labels_)
        return km.labels_, silhouettes
    
    def _consensus_arbiter(self, labels_list, confidences_list, weights):
        # Weighted voting per sample
        n_samples = len(labels_list[0])
        joint_labels = np.zeros(n_samples, dtype=int)
        consensus_confs = np.zeros(n_samples)
        
        for i in range(n_samples):
            votes = {}
            for modality_idx in range(3):
                label = labels_list[modality_idx][i]
                conf = confidences_list[modality_idx][i]
                votes[label] = votes.get(label, 0) + conf * weights[modality_idx]
            
            joint_labels[i] = max(votes, key=votes.get)
            consensus_confs[i] = max(votes.values()) / sum(votes.values())
        
        return joint_labels, consensus_confs
    
    def _detect_conflicts(self, label_V, label_I, label_T):
        conflicts = []
        for i in range(len(label_V)):
            labels = [label_V[i], label_I[i], label_T[i]]
            if len(set(labels)) > 1:  # Not all same label
                conflicts.append(True)
            else:
                conflicts.append(False)
        return np.array(conflicts)
    
    # DIRECTION 2
    def direction2_physics_constraint_filter(self, centroids, constraint_params):
        # Denormalize and test constraints
        valid_clusters = []
        violations_log = {}
        
        for cluster_id, centroid_norm in centroids.items():
            centroid_phys = self._denormalize(centroid_norm)
            violations = []
            
            # Test Joule heating
            T_max = constraint_params['T_ambient'] + \
                    (centroid_phys['I']**2 * constraint_params['R']) / \
                    (constraint_params['h'] * constraint_params['A'])
            if centroid_phys['T'] > T_max + 5:
                violations.append('Joule_heating')
            
            # Test vibration bounds
            if centroid_phys['a'] > constraint_params['a_max']:
                violations.append('Vibration_bounds')
            
            # Test efficiency
            if self._compute_efficiency(centroid_phys) > constraint_params['eta_nom'] + 0.05:
                violations.append('Power_efficiency')
            
            # Test cross-modal coherence
            # (requires cluster data, not just centroid)
            
            if not violations:
                valid_clusters.append(cluster_id)
            else:
                violations_log[cluster_id] = violations
        
        return valid_clusters, violations_log
    
    def _denormalize(self, centroid_norm):
        return {
            'T': centroid_norm[0] * self.eq_params['T_max'],
            'I': centroid_norm[1] * self.eq_params['I_max'],
            'a': centroid_norm[2] * self.eq_params['a_max']
        }

CLAIM 1 (Main):

A computer-implemented method for unsupervised discovery of physically-
feasible operational modes in cyber-physical systems, comprising:

(a) receiving continuous multimodal sensor data from a plurality of 
    heterogeneous sensors (vibration, electrical current, temperature)
    mounted on a physical system;

(b) performing per-modality unsupervised clustering by:
    (i) extracting modality-specific features from each sensor stream 
        independently (vibration features ≠ current features ≠ temp features)
    (ii) applying k-means clustering to each modality's feature set, 
         yielding per-modality cluster labels and silhouette confidence scores
    (iii) maintaining per-modality cluster assignments as independent 
          data structures (not pre-concatenated);

(c) performing weighted consensus arbitration by:
    (i) for each time point, combining per-modality labels via weighted voting
        where weights are derived from per-modality silhouette confidence scores
    (ii) producing a joint operational mode label that represents consensus
         across all modalities
    (iii) computing a consensus confidence score indicating degree of 
          modality agreement;

(d) identifying modality-level disagreement when per-modality labels diverge 
    beyond a threshold and generating a modality conflict flag as an explicit 
    system output;

(e) validating discovered modes against physical feasibility by:
    (i) denormalizing each mode centroid from feature space back to physical units
    (ii) evaluating the denormalized centroid against a set of parametric 
         physics constraint inequalities derived from equipment design 
         parameters, the inequalities encoding conservation laws 
         (Joule heating: T ≤ T_ambient + I²R/(hA),  
          Power balance: η ≤ η_nominal + ε,  
          Thermal inertia: |dT/dt| ≤ system_rate,  
          Vibration bounds: a ≤ a_max,  
          Cross-modal coherence: corr(dI/dt, dT/dt) ≥ threshold)
    (iii) classifying each mode as PHYSICALLY_COHERENT (all constraints satisfied) 
          or PHYSICALLY_INCOHERENT (any constraint violated);

(f) logging constraint violations as diagnostic metadata, each violation 
    encoded with root-cause attribution (Joule heating violation → electrical fault,  
    thermal inertia violation → sensor malfunction, efficiency violation → 
    measurement error);

(g) computing a modality dominance signature for each discovered joint mode, 
    the signature representing the fractional contribution of each modality's 
    clustering confidence to final mode identification, indicating which 
    sensor modality is most diagnostic for that mode;

(h) outputting the discovered and validated operational mode labels, together 
    with modality conflict flags, constraint violation logs, and modality 
    dominance signatures to an external system or operator interface.

## PART 11: QUICK REFERENCE FOR LLM AGENTS

### 11.1 Project Summary (One Paragraph)

We are building an unsupervised operational mode discovery system that analyzes multimodal sensor data (vibration, current, temperature) from physical equipment to identify distinct behavioral regimes WITHOUT predefined labels. The baseline approach (feature extraction → k-means clustering → validation) is NOT patentable because IBM patent US11277425B2 directly anticipates it. To achieve patentability, we implement Direction 2 (physics constraint filtering) and Direction 1 (per-modality consensus architecture), creating a novel two-tier validation approach: first, consensus arbitration across independently-clustered modalities with conflict detection; second, filtering of discovered modes through parametric physics constraints (Joule heating, thermal inertia, power balance, vibration bounds, cross-modal coherence). This combination is novel, non-obvious, and defensible under 35 USC §102 and §103.

### 11.2 Three Directions at a Glance

**Direction 1 — Hierarchical Consensus (Architectural Novelty)**
- What: Per-modality clustering maintained independently until consensus
- Why Novel: No prior art tracks per-modality silhouette confidence through consensus
- Patent Type: Architectural claim on hierarchical multi-level system
- Effort: 2 weeks, Medium difficulty
- Novelty: ⭐⭐⭐⭐⭐

**Direction 2 — Physics-Constrained (Constraint-Based Novelty)**
- What: Parametric physics inequalities filter discovered clusters
- Why Novel: No prior art merges unsupervised clustering with equation-based constraints
- Patent Type: Method claim on constraint-guided validation
- Effort: 2 weeks, Medium difficulty
- Novelty: ⭐⭐⭐⭐⭐ (HIGHEST)

**Direction 3 — Adaptive Data-Rate (Data-Flow Novelty)**
- What: Sampling rate dynamically switches based on system stability
- Why Novel: No prior art adapts sensing architecture itself, nor uses reconstruction error as feedback
- Patent Type: Method claim on adaptive-rate acquisition + manifold prediction
- Effort: 4 weeks, Hard difficulty
- Novelty: ⭐⭐⭐⭐

### 11.3 Why NOT Direction 3 as Primary?

Direction 3 is novel but harder to implement and explain. Direction 2 is more defensible (physics ≠ obvious) and clearer to patent examiners (constraint-based validation is a well-understood concept). Combine Direction 2 + 1 for primary filing, then add Direction 3 as continuation-in-part (CIP) later.
